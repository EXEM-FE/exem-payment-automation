# 09. 로드맵 / 구현 결과

## 1.5차 — "PC 인라인 편집 + 룰 보강" (2026-05-07 ✅)

1차 MVP 위에 4가지 문제 해결:

### 보강 항목
- ✅ **명세서 = SSoT 모델 도입**: 명세서 1행 = 표 1행. 매칭 안 된 row는 자동 시드 entry로 채워 행 수 보존
- ✅ **Step 3 인라인 편집 표** (카드 → 표): 명세서 14컬럼 readonly + 사용자 입력 6컬럼 (계정/함께한 사람/업무/신청/개인/영수증). 가로 스크롤. 자체 구현 (TanStack Table 미도입)
- ✅ **휴리스틱 매칭**: `normalizeMerchant` + Sørensen-Dice bigram 유사도. 가맹점 표기 차이(PG 접두사 등) 자동 흡수
- ✅ **인당 한도 검증** (복리후생비 야근/휴일만): description에 "야근|야간" → 12,000/인, "휴일|주말" → 15,000/인. 출처: 2026년 경비규정 §II-6
- ✅ **영수증 다중 사진 popover**: 한 가맹점(=명세서 1행)당 N장 첨부 가능. 배민/쿠팡이츠 등 다중 영수증 시나리오 지원. PC에서도 직접 업로드 (메모리 Map)
- ✅ **검증 톤 일원화 — 모두 warn**: 영수증 누락/동반인 누락/50만 초과/인당 한도 모두 다운로드 차단 X. 검토 화면과 완료 화면에서 안내로만 노출
- ✅ **Step 2 "이전" 버그 fix**: onBack에서 statement state 모두 리셋 + drop-zone input value 매번 리셋(같은 파일 재선택 가능)
- ✅ **⋯ "입력 초기화"**: 부착된 entry를 자동 시드 상태로 되돌림 (명세서 행은 항상 보존)

### 아키텍처 변경
- `packages/shared/src/rules.ts`: `EntryIssue.type`에 `per_person` 추가
- `apps/web/src/data.ts`: `normalizeMerchant`, `merchantSimilarity`, `buildSeedEntry`, `seedAttachedEntries` 추가. `buildMatches` 5단계 우선순위로 재작성
- `apps/web/src/App.tsx`: `MatchStep`/`MatchCard` 제거 → `MatchTable` + `MatchRow` + `StatusDot` + `ChipCell` + `ReceiptCell` + `ReceiptPopover` + `RowMenu`
- `apps/web/src/styles.css`: 표 스타일, 셀 위젯, 영수증 popover 스타일 추가

## 1차 MVP — "야근 식대 자동화" (✅ 동작)

목표는 FE1팀 ~10명이 11월 경비를 이 앱으로 작성·다운로드까지 완주하는 것이었고, 1차 스코프 항목이 모두 동작 가능한 상태가 되었다.

### 구현된 기능

#### 모바일 PWA
- ✅ 온보딩 (부서·이름 + 9명 토글 칩, localStorage)
- ✅ 저널 홈 (이번 달 항목 목록, 합계, 보강 배지)
- ✅ 빠른 추가 / 편집 (Bottom Sheet, 카드 탭으로 같은 모달 재사용)
  - 카메라/갤러리 다중 사진 (`OffscreenCanvas`로 1600px JPEG q=0.85 압축, IndexedDB 저장)
  - 일자, 금액, 가게 메모
  - **계정 자동 추천** (가맹점 정규식 패턴)
  - **식음료 4지선다** (가게가 식음료면 자동 노출)
  - **9명 토글 칩** (TEAM_MEMBERS 상수)
  - 본인 이름은 새 항목 작성 시 자동 활성
  - 영수증 필수/식대성 동반인 누락 인라인 경고
  - 임시저장 / 저장 / 삭제
- ✅ PC로 보내기 → 백엔드 push → PIN 4자리 발급 모달 (1h 카운트다운)
- ✅ 설정 (정보 수정, 이번 달 초기화)
- ✅ PWA manifest + SVG 아이콘(any/maskable) + Service Worker (오프라인 셸 캐싱)

#### PC 정산 콘솔
- ✅ 부서·이름 + PIN 4자리(input-otp)로 백엔드 pull
- ✅ 명세서 .xls 업로드·파싱 (브라우저 메모리)
- ✅ 자동 매칭 (날짜+금액 우선, 같은 날만 일치하면 검토)
- ✅ 매칭 카드 UI (자동/검토/누락 색깔 구분)
- ✅ 명세서에만 있는 행 → 가맹점 패턴 기반 계정 추천 + 새 항목 인라인 추가
- ✅ 50만원 초과 경고
- ✅ 영수증 필수 미첨부 / 동반인 누락 안내 (다운로드 차단 X)
- ✅ ExcelJS .xlsx 출력
  - 시트 1 (이용내역명세서): 14컬럼 + 사용자 추가 6컬럼 + 헤더 합계
  - 시트 1-1 (지출증빙 첨부): 8개 슬롯 merged + 사진 anchor (`SLOT_BOXES`)
  - 시트 2 (후불교통, 비목표 안내) + 예시
  - 파일명: `(카드)제경비신청서_M월_부서_이름.xlsx`
- ✅ 다운로드 후 자동 `DELETE /api/me` → 서버 슬롯 즉시 삭제
- ✅ 완료 화면 + "새로 시작"

#### 백엔드 (Bun + Hono / Vercel API)
- ✅ 메모리 `Map<userKey, Slot>` 단 하나
- ✅ `/api/push` (multipart), `/api/pull`, `/api/photos/:id`, `DELETE /api/me`
- ✅ TTL 정리 인터벌 (60초), 사용자당 30MB / 전역 1GB 한계
- ✅ PIN 1회용 + 1h, slot 24h
- ✅ pullToken 발급 후 PIN 즉시 폐기
- ✅ 정적 PWA 산출물 동시 서빙 (단일 프로세스)
- ✅ End-to-end smoke 테스트 통과 (push → pin → pull → 1회용 검증 → delete → 404)

#### 룰 엔진
- ✅ `rules/2026-current.json` (+ `packages/shared/rules.json`)
- ✅ 14개 계정 enum
- ✅ merchant_rules (주유·교통·통신·도서·사무용품·편의점·마트)
- ✅ food_merchant_patterns / receipt_required_patterns
- ✅ 한도 (50만원 품의서, 회식비 월 인당 등)

### 1차에서 단순화한 것 (설계 → 구현 시 결정)

- **자주 쓰는 멤버 묶음 / 자동완성 / 빈도 누적** → 모두 빼고 9명 토글 칩 단일 UI로 통일
- **사용자 명시적 모드 토글** → 제거. `pointer: coarse` + 폭 < 920 자동 분기
- **샘플 데이터 / 샘플 복구 버튼** → 제거. 깨끗한 첫 진입
- **Onboarding 화면** → 추가. profile 없으면 풀스크린 첫 진입
- **공유 템플릿 .xlsx 번들** → 코드(ExcelJS)로 from-scratch 생성으로 대체. 셀 매핑은 docs/07 그대로
