# 11. 공식 문서 기반 개선 후보

> 기준 문서: `docs/reference-pdfs/2026년 경비규정.pdf`, `docs/reference-pdfs/개인법인카드 경비신청방법.pdf`, `docs/reference-pdfs/경비 신청 유의사항.pdf`
>
> 작성일: 2026-05-12

## 이미 잘 맞는 부분

- 개인법인카드 청구명세서 업로드 후 `1. 이용내역명세서`를 채우는 흐름은 현재 `apps/web/src/App.tsx`와 `apps/web/src/xlsx.ts`에 구현되어 있다.
- 식대 한도(야근 12,000원, 휴일 15,000원), 50만원 이상 품의 안내, 영수증 누락 안내, 동반자 수와 업무 내용의 인원 불일치 안내는 `packages/shared/src/rules.ts`에 일부 반영되어 있다.
- 다운로드 후 DaouOffice에 옮겨 적을 합계, 경비신청금액, 개인사용금액, 사용 월, 사용 기간, 결재선 안내는 완료 화면에서 제공한다.

## 개선 우선순위

### P1. 영수증 필수와 권장 안내 분리

- 근거: `경비 신청 유의사항.pdf` 9~10p, `개인법인카드 경비신청방법.pdf` 16~17p.
- 문서 요구: 커피전문점, 가맹점명만으로 상세 내역 확인이 어려운 곳, 인터넷쇼핑몰, 편의점, 마트, 백화점, 문구점, 해외 가맹점, 배달 앱, 숙박 앱은 증빙 첨부가 필요하다. 해외 숙박은 영수증과 상세 인보이스를 모두 요구한다.
- 현재 상태: `packages/shared/rules.json`의 `receipt_required_patterns`는 일부 가맹점군만 포함하고, `validateForExport`는 모든 이슈를 안내로만 취급한다.
- 개선안: `required_receipt_patterns`, `recommended_receipt_patterns`, `invoice_required_patterns`처럼 강도를 나누고, 완료 전 검토 화면에서 "규정상 필수"와 "확인 권장"을 다른 톤으로 표시한다.

### P1. 식대 사용자/타부서 검증 강화

- 근거: `개인법인카드 경비신청방법.pdf` 13p.
- 문서 요구: 식대(야근/점심/출장) 작성 시 모든 사용자를 기재해야 하며, 타 부서 인원 결제 건은 경비 인정 대상이 아니다.
- 현재 상태: `apps/web/src/data.ts`의 `TEAM_MEMBERS`는 FE1팀 9명 고정이고, 검증은 인원 수/내용 불일치 중심이다.
- 개선안: 참여자에 부서 정보를 붙이거나, 최소한 "타부서/외부 인원 포함" 표시를 별도 상태로 둔다. 식대 항목에는 모든 사용자 입력 여부를 더 강하게 확인하고, 타부서 포함 시 개인사용금액 또는 신청 제외 후보로 안내한다.

### P1. 계정과목 원본 정합성 확인

- 근거: `개인법인카드 경비신청방법.pdf` 11p.
- 문서 요구: 계정 예시는 마지막 항목을 `선급금(임직원 개인사용 액)`으로 설명한다.
- 현재 상태: `packages/shared/src/types.ts`, `packages/shared/rules.json`, `rules/2026-current.json`은 14번째 계정으로 `잡비`를 사용한다.
- 개선안: 실제 신청서 템플릿의 드롭다운/양식이 `선급금`인지 `잡비`인지 확인한 뒤, 계정 enum과 문서 카탈로그를 하나로 맞춘다. 개인사용액을 별도 컬럼으로 이미 계산하므로 `선급금`을 사용자 선택 계정으로 노출할지도 함께 결정해야 한다.

### P1. 주차/통행료 현금 처리 안내

- 근거: `2026년 경비규정.pdf` 2p, `경비 신청 유의사항.pdf` 18p.
- 문서 요구: 카드 결제가 어려운 통행료와 주차요금은 현금 사용 후 일반경비로 신청한다.
- 현재 상태: 여비교통비 계정은 있지만 주차/통행료를 카드 신청 흐름에서 별도 검증하지 않는다.
- 개선안: 가맹점/업무 내용에 주차, 주차장, 톨, 통행료, 하이패스가 있으면 "카드 신청 대상인지 확인" 안내를 띄운다. 일반경비로 돌려야 하는 경우에는 신청서에 넣지 않도록 제외 옵션을 제공한다.

### P1. 50만원 이상 사전 품의/지출결의 흐름 구체화

- 근거: `경비 신청 유의사항.pdf` 2~4p, `2026년 경비규정.pdf` 10p.
- 문서 요구: 규정 외 50만원 초과 비용은 사전 품의가 원칙이고, 접대성 경비는 건당 부가세 포함 50만원 이상이면 별도 지출결의서가 필요하다.
- 현재 상태: `validateForExport`는 50만원 이상을 단일 안내로만 처리하고, `접대성_별도결의서_기준`, `외부지원접대_사전승인`은 런타임 검증에 충분히 쓰이지 않는다.
- 개선안: 50만원 초과 일반 항목, 접대성 50만원 이상, 외부지원 접대 10만원 이상을 서로 다른 확인 항목으로 분리한다. 품의서 번호/작성 여부를 사용자가 기록할 수 있는 필드를 추가하면 월말 반송을 줄일 수 있다.

### P2. 주유비 계산/월 귀속 지원

- 근거: `개인법인카드 경비신청방법.pdf` 14p, `경비 신청 유의사항.pdf` 12p.
- 문서 요구: 주유비는 `거리(km) * 편도/왕복 * 이용일수`로 기재하고, 1km당 300원을 적용한다. 해당 월 외근 주유비는 같은 달에 결제해야 하며 다음 달 이월 결제는 불가하다. 주유 할인 금액은 여비교통비로 음수 신청금액을 기재한다.
- 현재 상태: 주유 가맹점 계정 추천은 있지만 거리 계산, 월 귀속 확인, 음수 할인 행 처리는 별도 UX가 없다. `parseStatementFile`은 청구금액이 0보다 큰 행만 남기므로 음수 할인 행이 빠질 수 있다.
- 개선안: 주유비 프리셋을 추가해 거리/편도·왕복/일수를 입력하면 신청금액과 업무 내용을 자동 생성한다. 명세서 파싱은 음수 할인 행 보존 여부를 테스트로 고정한다.

### P2. 회식비 월 한도/이월 안내

- 근거: `2026년 경비규정.pdf` 6p, `경비 신청 유의사항.pdf` 8p.
- 문서 요구: 회식비는 매월 1인당 50,000원이며 미사용분은 이월 가능하지만 다음 달 분 선사용은 불가하고, 12월 말 미사용분은 소멸된다.
- 현재 상태: `회식비_월_인당` 값은 룰 JSON에 있으나 월 누적/이월 계산은 현재 비목표로 남아 있다.
- 개선안: 자동 계산이 어렵다면 먼저 "이번 달 회식비 사용액/예상 한도" 로컬 요약을 제공한다. 12월에는 소멸 안내를 별도로 띄운다.

### P2. 우편/택배 계정 추천 보강

- 근거: `경비 신청 유의사항.pdf` 8p.
- 문서 요구: 일반/등기 우편은 `통신비`, 택배/해외택배는 `운반비` 계정을 사용한다.
- 현재 상태: `merchant_rules`는 통신사/교통/도서/사무용품/편의점/마트 중심이고, 우편/택배 분류 패턴이 약하다.
- 개선안: 우체국, 등기, 우편, 택배, 퀵서비스, 해외배송 키워드를 계정 추천 룰에 추가한다.

### P2. DaouOffice 기안 단계 체크리스트 확장

- 근거: `개인법인카드 경비신청방법.pdf` 22~36p.
- 문서 요구: 법인카드 경비신청서 양식 선택, 사용 월/사용 기간 입력, 신청서 첨부, 합계/경비신청금액/개인사용금액 입력, 결재 정보 확인, 반송 시 재기안을 수행한다.
- 현재 상태: 완료 화면은 그룹웨어 링크와 복사 가능한 값은 제공하지만, 첨부 여부, 문서 제목, 재기안, 결재 여부 확인은 안내하지 않는다.
- 개선안: 완료 화면을 제출 체크리스트로 확장한다. 실제 DaouOffice API 연동 전이라도 "파일 첨부 완료", "합계 입력 완료", "결재 요청 완료" 같은 로컬 체크 상태를 둘 수 있다.

### P3. 후불교통/하이패스 시트 자동화

- 근거: `개인법인카드 경비신청방법.pdf` 18~21p.
- 문서 요구: 후불교통/하이패스 이용내역을 별도 조회하고, 택시/고속버스 내역을 삭제한 뒤 후불교통비 이용명세서에 붙여넣는다.
- 현재 상태: 현재 스코프에서는 후불교통 시트를 적극적으로 채우지 않는다.
- 개선안: 실제 사용자가 후불교통 데이터를 갖고 오기 시작하면 두 번째 업로드 입력을 열고, 택시/고속버스 제외 규칙과 시트 2 매핑을 추가한다.

### P3. 룰 JSON 단일화

- 근거: 현재 레포 구조.
- 현재 상태: 앱은 `packages/shared/rules.json`을 직접 import하고, `rules/2026-current.json`은 같은 내용을 별도로 보관한다.
- 개선안: `rules/2026-current.json`을 원본으로 삼아 빌드 때 `packages/shared/rules.json`으로 복사하거나, 앱이 원본 경로만 import하도록 정리한다. 규정 PDF가 갱신될 때 한 파일만 바꾸게 만드는 것이 핵심이다.
